# Frontend Payments Architecture (React & CRM Integration)

This document serves as the client-side reference for the payment integrations, covering the public Next.js interfaces and the custom React Admin CRM logic.

## 1. Checkout Flows

### Online Lessons (`LessonRequestComponent.tsx`)
The `LessonRequestComponent` acts as a dynamic pricing calculator and automatic checkout gateway.
*   **State Calculation:** Utilizes complex internal state to evaluate selected course fields, intended duration, number of students, and package multipliers (single vs. bundle).
*   **Dynamic Discounts:** Processes localized pricing tiers to determine exact amounts dynamically before committing to the database.
*   **Flow:** Upon submission, the frontend POSTs the finalized application payload. It immediately intercepts the returned application ID and initiates a secondary POST to `/api/payments/checkout`. It then gracefully extracts the resulting `formUrl` and redirects the client to the bank's hosted page.

### Public Invoice Landing Page (`pages/checkout/service.tsx`)
This page handles manual invoices generated by administrators for custom service requests.
*   **Security:** To prevent parameter tampering, the URL only accepts an `?id=` token (the MongoDB `Payment` ID).
*   **Data Hydration:** Fetches the immutable payment details from the backend (`GET /api/payments/public/:id`), displaying a summary containing the customer's name, requested service details, and the locked `amountAMD`.
*   **Action:** Displays a localized "Pay Now" button. Clicking this button triggers `POST /api/payments/initiate-service` to securely fetch a fresh `formUrl` from the bank without risking a token expiration timeout.

## 2. Callback Handling

### Unified Result Page (`pages/payment-result.tsx`)
Because InecoBank historically utilizes different callback architectures depending on the payment origin or error state, the callback URL parsing must be highly robust.
*   **Parameter Verification:** The bank returns query tokens asynchronously. The callback handler searches for `mdOrder`, `orderId`, and `paymentID` to isolate the transaction token.
*   **Resolution:** Dispatches a validation check to `GET /api/payments/verify/:token`.
*   **Localization (i18n):** Intercepts the response (`COMPLETED` or `FAILED`) and maps the result to the localized `invoice` / `payment` string dictionaries located in `en.json`, `ru.json`, and `hy.json`.

## 3. Custom CRM Integration (React Admin)

### Managing Pricing Arrays
The platform incorporates deep nested configuration menus to control the dynamic checkout pricing structures.
*   **Architecture:** Uses React Admin's `<ArrayInput>` and `<SimpleFormIterator>` structures to allow staff to add infinite rows to the `pricingSections`, configuring duration matrixes and discount rules natively.

### Contextual Payment Links (`PaymentsList.tsx` & `PaymentsShow.tsx`)
Because the `Payment` collection serves dual purposes (both Courses and Services), the CRM cannot use a hardcoded `<ReferenceField>`.
*   **Dynamic References:** Implements a custom `DynamicApplicationReference` component utilizing React Admin's `useRecordContext()`. 
*   **Routing Logic:** Inspects the current record. If `serviceApplicationId` is populated, it renders a reference pointing to `services-application-request`. If `applicationId` is populated, it falls back to `courses-application-request`.
*   **UX Enhancement:** Injects the `link="show"` property to ensure that clicking the parent application bypasses the Edit form and opens the read-only Show view directly.

### Defensive Rendering (Orphan Prevention)
In the event that an administrator deletes a Course Application or Service Application from the database, the related Payment log persists for audit reasons. 
*   **Bug Prevention:** If React Admin attempts to fetch a reference that has been destroyed, its `simple-rest` provider historically throws a fatal "Element does not exist" error, breaking the entire Datagrid.
*   **Fix:** All dynamic reference fields are strongly typed with the `emptyText="Deleted / Missing"` prop, ensuring that the CRM fails gracefully and simply prints the text rather than crashing the interface.
